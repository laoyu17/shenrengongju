## 2. 发现清单（按严重级别）

> 说明：本轮未发现“阻断上线的致命运行时 bug（P0）”；发现 3 项语义/治理层高优先级问题（P1）与 1 项低优先级问题（P2）。

### F-01（P1）研究报告可出现“总体 fail 但失败条目为空”，不利于定位
- 现象：`research_report` 的 `status` 由 `audit/model_relations/quality` 联合判定，但 `failed_check_details` 仅汇总审计规则失败。
- 结果：当 `model_relations=warn/fail` 且审计规则全通过时，会出现“`status=fail` 但 `failed_check_details=[]`”。
- 证据：
  - 总体状态判定：`rtos_sim/analysis/research_report.py:131`
  - 失败详情来源仅 audit checks：`rtos_sim/analysis/research_report.py:143`
  - 复现输入示例：`examples/at10_arrival_process.yaml`（本地复现结果）
- 风险：评审报告可读性下降，失败原因不透明，人工排查成本上升。

### F-02（P1）CI 研究报告样本固定为单一场景，覆盖不足
- 现象：`research_audit` job 的评审报告构建仅使用 `examples/at02_resource_mutex.yaml`。
- 证据：`.github/workflows/ci.yml:152` 与 `.github/workflows/ci.yml:157`
- 风险：CI Summary 可能长期“绿色”，但未覆盖 `inspect-model` 在其他官方样例中的 warn/fail 情况。
- 补充现象：本地抽查中 5/10 官方运行样例 `model_relations.research_v1=fail`。

### F-03（P1）模型关系 profile 与规则严重度存在语义张力
- 现象：`segment_core_binding_coverage` 定义为 `severity=warn`，但被纳入 engineering/research profile 必过项，导致 profile 直接 fail。
- 证据：
  - profile 必过项定义：`rtos_sim/analysis/model_relations.py:28`
  - 规则严重度定义：`rtos_sim/analysis/model_relations.py:116`
  - profile 失败条件：`rtos_sim/analysis/model_relations.py:54`
- 风险：同一份报告中可能出现 `status=warn` 但 `profile=fail`，业务语义解释成本较高。

### F-04（P2）`inspect-model` CLI 返回码不反映语义失败
- 现象：`cmd_inspect_model` 在报告生成成功时始终返回 0，不区分 `status=pass/warn/fail`。
- 证据：`rtos_sim/cli/main.py:294`、`rtos_sim/cli/main.py:320`
- 风险：自动化流程若不额外解析 JSON，无法用退出码快速失败。

## 6. 季度路线图（不含性能优化）

### 阶段 Q1（0–4 周）：语义闭环口径收敛
1. 让 `research_report` 输出“非 audit 失败原因明细”（覆盖 model_relations/quality 失败来源）。
2. 统一模型关系 profile 与 severity 语义（至少明确“warn 是否阻断 profile”规则）。
3. 增加对应单元测试（新增“report fail but empty details”回归测试）。

**验收**：不存在“`status=fail` 且 `failed_check_details=[]` 且无 warnings”的报告。

### 阶段 Q2（4–8 周）：CI 研究审查代表性提升
1. `research_audit` 从单样例扩展为“最小多样例集”（至少覆盖：有 unbound 的样例 + 资源绑定样例 + 时间确定性样例）。
2. Step Summary 同时展示 `audit.research_v1` 与 `model_relations.research_v1` 聚合状态。
3. 明确非阻断告警阈值与处理 SOP。

**验收**：CI 报告可反映多样例真实语义，不再依赖单样例。

### 阶段 Q3（8–12 周）：文档与验收契约固化
1. 在 `docs/14/15` 固化统一判定口径（audit 与 inspect-model 的关系）。
2. 将“研究评审失败分类”写入模板报告字段规范。
3. 建立发布前 checklist（纯语义与可追溯性，不含性能项）。

**验收**：双基线文档、脚本、CI summary 三者口径一致。
