## 1. 工具能做什么

`rtos-sim` 是一个配置驱动的 RTOS 异构多核调度仿真工具，支持：
- 配置合法性校验（Schema + 语义）
- 仿真运行与事件/指标导出
- 审计报告导出（协议一致性/研究闭环）
- 模型关系导出（任务/子任务/分段 与 核/资源）
- 批量实验（参数矩阵）
- 指标对比（JSON + CSV）
- 可选 PyQt UI 可视化

## 5. CLI 命令详解

### 5.1 `validate`（配置校验）

用途：检查配置文件结构与语义是否合法。

```bash
rtos-sim validate -c <config.yaml>
```

严格 ID 分隔符模式（建议 CI 用）：

```bash
rtos-sim validate -c <config.yaml> --strict-id-tokens
```

注意：`examples/batch_matrix.yaml` 属于批跑配置，不可用 `validate`。

---

### 5.2 `run`（运行仿真）

用途：执行仿真并输出事件、指标、审计报告。

```bash
rtos-sim run -c <config.yaml> \
  --events-out artifacts/events.jsonl \
  --metrics-out artifacts/metrics.json \
  --audit-out artifacts/audit.json
```

常用可选参数：
- `--until <float>`：覆盖仿真时长
- `--step --delta <float>`：步进执行
- `--pause-at <float>`：在指定仿真时间暂停并导出部分结果
- `--events-csv-out <path>`：导出事件 CSV（用于表格分析）

失败判定：
- 配置错误返回 `1`
- 审计失败（`--audit-out` 且 audit 非 pass）返回 `2`

---

### 5.3 `inspect-model`（模型关系导出）

用途：导出任务/子任务/分段与核/资源关系，并给出语义判定。

```bash
rtos-sim inspect-model -c <config.yaml> \
  --out-json artifacts/model_relations.json \
  --out-csv artifacts/model_relations.csv
```

严格门禁模式：

```bash
rtos-sim inspect-model -c <config.yaml> --strict-on-fail
```

失败判定：`status != pass` 时返回 `2`（仅 strict 模式）。

---

### 5.4 `batch-run`（批量实验）

用途：按矩阵展开多组实验并汇总。

```bash
rtos-sim batch-run -b examples/batch_matrix.yaml
```

常用可选参数：
- `--output-dir <dir>`：指定批跑输出目录
- `--summary-csv <path>`：指定汇总 CSV 路径
- `--summary-json <path>`：指定汇总 JSON 路径

目录边界规则（2026-03-05 更新）：
- 批配置文件中的 `output_dir` 必须落在批配置所在目录内，防止 `../` 越界写文件。
- 如确需输出到工作区外，请使用 CLI 显式参数 `--output-dir <absolute-or-relative-path>`。

失败可观测性（2026-03-05 更新）：
- `summary.json` 的失败行现在包含 `error`、`error_type`、`error_trace_path`。
- 可直接打开 `error_trace_path` 对应文件定位完整 traceback。

严格失败模式（任一子任务失败即整体非 0）：

```bash
rtos-sim batch-run -b examples/batch_matrix.yaml --strict-fail-on-error
```

---

### 5.5 `compare`（指标对比）

用途：对比两份 metrics JSON，输出差分结果。

```bash
rtos-sim compare \
  --left-metrics artifacts/base_metrics.json \
  --right-metrics artifacts/new_metrics.json \
  --out-json artifacts/compare.json \
  --out-csv artifacts/compare.csv
```

输出重点：
- `scalar_metrics`：标量指标差分
- `core_utilization`：按核利用率差分

---

### 5.6 `migrate-config`（配置迁移）

用途：迁移旧配置并清理已废弃字段。

```bash
rtos-sim migrate-config \
  --in examples/at01_single_dag_single_core.yaml \
  --out artifacts/migrated.yaml \
  --report-out artifacts/migrate_report.json
```

仅迁移不校验：

```bash
rtos-sim migrate-config --in <old> --out <new> --no-validate
```

## 6. UI 操作指南（基础流程）

1. **加载配置**：打开示例 YAML（建议先用 `at01`）。
2. **编辑配置**：
   - 可在结构化表格中编辑任务/资源；
   - 可在 DAG 区域增删节点/边并做自动布局。
3. **校验配置**：点击 Validate（或先走 CLI `validate`）。
4. **运行仿真**：启动后观察 Gantt 与日志。
5. **查看详情**：悬停/点击线段查看任务、分段、核心等详情。
6. **导出结果**：保存 YAML，并通过 CLI 导出 events/metrics/audit。

提示：UI 不是真相源，最终审查请以 CLI 导出的 JSON/JSONL 为准。

## 8. 故障定位与排查矩阵（推荐按顺序）

| 现象 | 快速判定 | 处理建议 |
|---|---|---|
| `validate` 校验 `batch_matrix.yaml` 报错 | 输入是批跑配置，不是运行配置 | 改用 `rtos-sim batch-run -b examples/batch_matrix.yaml` |
| `run --audit-out` 返回码 2 | `audit.status != pass` | 打开 `audit.json`，优先看 `failed_checks` / `issues` |
| `inspect-model --strict-on-fail` 返回码 2 | 关系报告 `status != pass` | 打开 `model_relations.json`，逐条修复 `checks` |
| `batch-run --strict-fail-on-error` 返回码 2 | 至少 1 个子案例失败 | 看 `summary.json` 的 `runs[*].error` 并单例复跑 |
| `batch-run` 提示 `output_dir escapes batch workspace` | 批配置中的 `output_dir` 越界（如 `../`） | 改为批目录内路径，或使用 CLI `--output-dir` 显式指定 |
| UI 启动失败 | 缺 UI 依赖或图形库 | 安装 `.[ui]`，Linux 补系统库 |
| Wayland 下出现 `Failed to create popup` 或 UI 崩溃 | tooltip popup transient parent 不完整（旧版本） | 升级到包含 `fix(ui): avoid Wayland tooltip popup parent crash` 的版本；临时可用 `QT_QPA_PLATFORM=xcb` |

## 10. 推荐工作流（团队统一）

1. 开发/测试先跑 `validate`。
2. 变更后跑 `run --audit-out`。
3. 变更模型结构时补跑 `inspect-model --strict-on-fail`。
4. 合并前执行 `pytest` 和最小命令级回归。
5. 评审材料统一引用 `artifacts/` 产物。

## 11. 命令级决策树（文本版）

```text
开始
 ├─ 先跑 validate
 │   ├─ rc=0 -> 继续 run
 │   ├─ rc=1 -> 修配置语法/语义
 │   └─ rc=2 -> 处理 ID 分隔符风险（strict 模式）
 ├─ 再跑 run --audit-out
 │   ├─ rc=0 -> 继续 inspect-model
 │   ├─ rc=1 -> 看运行错误（配置/运行时）
 │   └─ rc=2 -> 看 audit.failed_checks，修规则失败项
 ├─ 跑 inspect-model --strict-on-fail
 │   ├─ rc=0 -> 可进入 compare / batch-run / 评审
 │   └─ rc=2 -> 看 model_relations.checks，修语义映射
 └─ 跑 batch-run --strict-fail-on-error（如需要）
     ├─ rc=0 -> 批跑通过
     └─ rc=2 -> 从 summary.json 找失败 case，单例复跑
```

## 14. `legacy/report_api` 兼容层语义边界（I2-2）

`rtos_sim.legacy.report_api` 的定位是“报告同名接口兼容层”，用于保证表 6-2 同名函数可调用；它不是运行时调度器的完整替代。

边界约定：
- `sched_schedule`：只基于 `schedule_table` 在 `now` 时刻做窗口查询，不推进仿真时钟、不维护 ready/running 队列。
- `sched_td_task_new_arrival` / `sched_td_task_complete` / `sched_dy_task_new_arrival` / `sched_dy_task_complete`：只维护轻量 `events/last_event` 历史字典，不触发引擎状态流转。
- `sched_model_change`：只返回配置补丁（如 `static_window_mode/static_windows`），不直接修改运行时引擎对象。

使用建议：
- 需要运行时等价语义（含抢占、资源协议、状态机）时，统一走 `rtos-sim run` 或 `rtos_sim.api` 主链路。
- 兼容层仅用于报告同名接口适配与轻量脚本封装。
