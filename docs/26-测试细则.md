# 测试细则（严格 Docx 收口）

## 1. 测试范围

- CLI 规划链路：`plan-static` / `analyze-wcrt` / `export-os-config`
- 指标链路：`benchmark-sched-rate` API 与脚本
- UI 链路：Planning 面板、状态流、随机任务生成
- 文档链路：`docs/26-*` 交付件完整性

## 2. 环境与工具

- Python 3.10+
- `pytest`
- UI 测试：`QT_QPA_PLATFORM=offscreen`

## 3. UI 最小回归门禁（当前治理口径）

当前阶段先不设 UI 模块覆盖率硬百分比门槛；日常回归与 freeze 前核验，至少执行下面最小必测集，并要求全部通过：

```bash
QT_QPA_PLATFORM=offscreen pytest -q   tests/ui/test_dag_controller.py   tests/ui/test_dag_controller_additional.py   tests/ui/test_planning_controller.py   tests/ui/test_telemetry_controller.py   tests/ui/test_run_controller.py   tests/ui/test_compare_controller.py   tests/test_ui_compare_io.py
```

矩阵口径：

- DAG：至少覆盖加边成功、成环失败、空态选择重置、空子任务删除 no-op。
- Planning：至少覆盖成功渲染、非法配置失败、空值/无效线程边界渲染。
- Telemetry：至少覆盖空白 hover、空白 click、父节点回退、reset 清理。
- Run：至少覆盖启动成功、非法配置失败、step/timeout 边界、finished/failed/reset 缓存清理。
- Compare：至少覆盖 latest 空态提示、构建成功、文件加载失败、导出链路、JSON root 非对象失败。

说明：只有当这组最小必测集长期稳定、失败样本可解释后，才考虑把 UI 治理从“最小集合门禁”升级到更硬的覆盖率阈值。

## 4. 用例设计

1. 指纹输出
   - 输入：合法配置执行 `plan-static`
   - 预期：输出 JSON 含 `spec_fingerprint`

2. 严格计划匹配（WCRT）
   - 输入：A 配置生成计划，B 配置直接分析（默认 strict）
   - 预期：返回码非 0（2），错误信息包含“期望指纹#”

3. 严格计划匹配（OS 导出）
   - 输入：A 配置计划 + B 配置导出（默认 strict）
   - 预期：返回码 2

4. 双口径指标
   - 输入：执行 `benchmark-sched-rate`
   - 预期：报告同时含 `uplift` 与 `candidate_only_uplift`

5. UI 规划闭环
   - 步骤：Planning 页执行 Plan → WCRT → Export
   - 预期：3 张结果表均有数据

6. UI 四态展示
   - 输入：状态事件流 `Released/Ready/Executing/Blocked`
   - 预期：状态面板显示四态文本

7. 随机任务可复现
   - 输入：同 seed 连续两次生成
   - 预期：输出配置文本完全一致

## 5. 判定标准

- 所有新增测试通过。
- 严格模式错误返回码与错误信息符合约定。
- 双口径字段可由报告直接复核。
- UI 最小必测集合可单独执行，且覆盖成功 / 失败 / 空态三类回归。
