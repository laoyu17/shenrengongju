# 需求分析报告（严格 Docx 收口）

## 1. 背景与目标

当前项目核心算法链路已具备，但严格 Docx 验收缺口集中在三类：

1. 结果可信性：计划结果与模型配置缺少强绑定校验。
2. 指标口径：可调度率 uplift 需区分严格与兼容口径。
3. 交付闭环：UI 规划能力与文档交付件需完整可审查。

目标是在不引入性能优化任务的前提下，完成严格验收闭环。

## 2. 功能需求

1. `plan-static` 输出模型指纹字段 `spec_fingerprint`。
2. `analyze-wcrt` 与 `export-os-config` 支持 `--strict-plan-match`，严格模式下拒绝跨模型计划复用。
3. 可调度率报告同时输出：
   - 严格口径：`candidate_only_schedulable_rate`、`candidate_only_uplift`
   - 兼容口径：`best_candidate_schedulable_rate`、`uplift`
4. UI 新增 Planning 页，支持：规划、WCRT、OS 导出、随机任务生成。
5. UI 状态流显式展示 Released/Ready/Executing/Blocked。

## 3. 非功能需求

1. 向后兼容：原命令默认行为不被破坏；严格能力通过新增参数启用。
2. 可测试：新增能力需有自动化测试覆盖。
3. 可追踪：每项需求可映射到代码点、测试点和交付件。

## 4. 约束与假设

1. 基线模型按仓内 `250909-仿真工具-基础模型.docx` 对齐。
2. 本轮不做性能调优与性能指标扩展。
3. 文档交付先以 Markdown 作为审查基线。
